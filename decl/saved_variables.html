saved_variables
/decl/saved_variables

Singleton datum for holding saved variables info for a given type. Don't define one of those manually, and instead use the macros. Objects of this type expect the path of the entity they contain info on to have their full path in the datum subtype's path. (Ex: /mob/living -> /decl/saved_variables/mob/living )

Vars

base_typeUsed for types that inherit from another, even though the path doesn't represents that (AKA /obj inheriting from /atom/movable, and /atom/movable inheriting from /datum)
cachedSaved vars cached after being generated from the macros, since that involves a lot of proc calls
should_flattenWhether the type should be saved entirely as a json array.
type_pathThe type path of the object this saved_vars decl is for

Procs

get_saved_variablesThis is what should be used to retrieve the saved var list. It lazy init the cached list and returns it.
make_saved_variablesThis is called by the macro to override the proc and set the name of the newly added saved var. This shouldn't be called before all decl of this type are instantiated!!!
test_variablesWill test the instance of the given object if its the object we're holding the saved vars for. Will returns FALSE if something is wrong with the variables.
